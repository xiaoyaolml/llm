# Cursor Playbook: 代码审查

## 适用场景

- 审查 PR / 分支差异
- 审查单个模块改动
- 审查回归修复

## 推荐用法

先让 Cursor 不改代码，只做审查：

```text
+先不要修改代码。
+请审查当前改动，并按以下结构输出：
+1. 正确性问题
+2. 安全问题
+3. 并发与生命周期问题
+4. 性能问题
+5. 可维护性问题
+6. 测试缺口
+7. 严重级别排序
+```
+
## 审查顺序
+
+1. 正确性
+2. 崩溃 / 未定义行为 / 安全问题
+3. 并发与资源生命周期
+4. 协议 / 接口兼容性
+5. 性能与可观测性
+6. 风格与可维护性
+
## 必查项
+
+- 空输入 / 边界值 / 极端值
+- 错误路径与资源释放
+- 日志是否足够定位问题
+- public API 是否破坏兼容性
+- 测试是否覆盖新增行为
+
## 输出建议
+
+要求 Cursor 输出：
+
+- 问题位置
+- 风险原因
+- 建议修复方式
+- 是否建议阻塞合并
## 实战示例：互联网金融 C++ 后端

### 场景：审查撮合引擎价格匹配逻辑

```text
先不要修改代码。
请审查 match_engine.cpp 的当前 PR 改动，重点关注：
1. 价格比较精度是否正确（浮点 vs 定点）
2. 订单优先级（价格优先、时间优先）是否被破坏
3. 边界条件：空订单簿、相同价格、涨跌停价格
4. 是否引入新的锁竞争（撮合线程 vs 行情推送线程）
5. 错误路径是否正确释放资源
6. 按严重级别排序所有问题
```

### 场景：审查风控限额检查逻辑

```text
先不要修改代码。
请审查 risk_limit_checker.cpp 的改动：
1. 限额计算是否存在 rounding/overflow 风险
2. 并发场景下限额扣减是否原子
3. 限额恢复（撤单/拒单）路径是否正确
4. 极端输入（零价格、最大数量、负值）是否已处理
5. 日志是否足够定位限额触发原因
```
