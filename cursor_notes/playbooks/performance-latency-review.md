# Cursor Playbook: 性能 / 延迟 / 并发审查

## 适用场景

- 热路径优化
- 尾延迟升高
- 并发瓶颈排查
- 网络 / 数据库 / 内存热点分析

## 推荐提示词

```text
+先不要修改代码。
+请分析这个模块的性能与延迟风险，并按以下结构输出：
+1. 热路径
+2. CPU / 内存 / 锁 / I/O / 网络瓶颈
+3. 额外分配与拷贝点
+4. 队列积压与背压风险
+5. tail latency 风险
+6. 最小侵入优化方案
+7. benchmark / profiling 验证方案
+```
+
## 审查原则
+
+- 不凭感觉认定瓶颈，优先要求证据
+- 区分平均延迟与 P99/P999
+- 区分业务链路与基础设施链路
+- 区分一次性优化与长期可维护优化
+
## 必查项
+
+- 锁持有时间
+- 批处理策略
+- 缓存局部性
+- false sharing
+- 临时对象与格式化开销
+- 序列化 / 反序列化成本
+- 网络往返与重试放大
+- 数据库慢查询与连接池饱和
## 实战示例：互联网金融场景

### 场景：行情分发链路 P99 抖动

```text
先不要修改代码。
现象：行情分发服务 P99 从 50us 升到 500us，CPU 无明显变化。
请分析：
1. 行情接收→解码→分发的完整热路径
2. 是否在热路径新增了内存分配（std::string / vector resize）
3. 日志写入是否同步阻塞
4. 是否存在 false sharing（多线程写相邻 cache line）
5. 锁持有时间是否因新逻辑增长
6. 给出最小侵入诊断方案（perf/flamegraph/rdtsc 埋点）
```

### 场景：订单网关吞吐瓶颈

```text
先不要修改代码。
现象：订单网关 QPS 无法超过 50K/s，CPU 只用到 30%。
请排查：
1. 网络 IO 模型（epoll 边缘/水平触发、单线程/多线程 accept）
2. 序列化/反序列化是否有冗余拷贝
3. 连接池/线程池是否饱和
4. 系统调用频率（send/recv/malloc 热点）
5. 内核参数（SO_RCVBUF / tcp_nodelay / busy_poll）
6. 给出 3 个优化方向及预期收益
```
