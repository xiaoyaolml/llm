# 代码审查规则库

## 1. 正确性检查

### 1.1 逻辑错误
- [ ] 是否处理了所有边界条件（空输入、零值、极值）
- [ ] 数组/切片访问前是否检查索引范围
- [ ] 指针/引用是否在解引用前检查有效性
- [ ] 循环条件是否正确，有无死循环或提前退出
- [ ] 递归函数是否有正确的终止条件
- [ ] 数学运算是否考虑溢出/下溢（尤其 `int` 加减乘）
- [ ] 浮点比较是否使用合理的误差范围（不直接用 `==`）

### 1.2 错误处理
- [ ] 文件/网络/数据库操作是否检查错误返回值
- [ ] 内存分配是否检查是否为 `nullptr`
- [ ] STL 容器操作（如 `at()`）是否会抛异常并捕获
- [ ] 析构函数是否捕获异常（应避免抛异常）

### 1.3 数据类型与转换
- [ ] 隐式类型转换是否安全（尤其是 `int` → `char`、`float` → `int`）
- [ ] `void*` 强制转换前是否验证类型
- [ ] 位域/联合体/结构体内存布局是否明确指定（避免平台差异）
- [ ] 字符串到数值转换是否检查格式和范围

---

## 2. 安全性检查

### 2.1 注入漏洞
- [ ] 是否使用 `snprintf` 而非 `sprintf`（防止缓冲区溢出）
- [ ] 字符串拼接是否使用 `strncpy` + 手动终止（而非 `strcpy`）
- [ ] 文件路径拼接是否验证目录穿越（`../`）
- [ ] 外部输入是否进行合法性校验（正则、白名单）

### 2.2 内存安全
- [ ] 动态分配的内存是否全部释放（检查 `new`/`delete`、`new[]`/`delete[]` 配对）
- [ ] 是否存在悬垂指针（使用已释放内存）
- [ ] 是否存在野指针（使用未初始化指针）
- [ ] 数组访问是否越界（常见于 `i <= size` 错误）
- [ ] 字符串操作是否处理了空终止符（`\0`）

### 2.3 敏感信息
- [ ] 是否有硬编码的密钥、密码、Token
- [ ] 配置文件（如 `.env`、`config.json`）是否被误提交
- [ ] 日志/调试输出是否包含敏感信息（用户密码、密钥）
- [ ] 内存中敏感数据使用后是否清零（如 `memset`）

### 2.4 访问控制
- [ ] 文件/目录权限设置是否合理（避免 `chmod 777`）
- [ ] 是否限制对关键资源的访问（如数据库、API 端点）

---

## 3. 性能检查

### 3.1 时间复杂度
- [ ] 嵌套循环是否有冗余（可提取不变量）
- [ ] 查找操作是否可以用哈希/二分优化
- [ ] 是否重复计算可缓存的结果

### 3.2 内存使用
- [ ] 大对象是否传值而非传引用（避免拷贝）
- [ ] 字符串拼接是否用 `+=` 而非重复创建新字符串
- [ ] 容器大小是否预先 `reserve()` 避免多次重分配
- [ ] 递归深度是否过大（可改为迭代）

### 3.3 编译器优化
- [ ] 是否有 `O2`/`O3` 优化友好的写法
- [ ] 分支预测友好（高频路径放在 `if` 块）
- [ ] 避免在循环中调用虚函数（如可能）

---

## 4. 可读性与可维护性

### 4.1 命名规范
- [ ] 变量名是否有明确含义（避免 `a`、`b`、`tmp`）
- [ ] 命名风格是否一致（`camelCase` vs `snake_case`）
- [ ] 常量是否全大写（如 `MAX_SIZE`）
- [ ] 宏名是否全大写（避免与函数混淆）

### 4.2 结构清晰
- [ ] 函数长度是否合理（建议 <50 行）
- [ ] 单个函数职责是否单一
- [ ] 是否有过多的嵌套层次（建议 <3 层）
- [ ] 是否使用有意义的空行分隔逻辑块

### 4.3 注释与文档
- [ ] 复杂算法是否有必要注释
- [ ] 接口函数是否有文档（参数、返回值、异常）
- [ ] 魔法数字是否替换为命名常量

---

## 5. 语言特定规则

### 5.1 C++ 核心规则
- [ ] 是否优先使用 `std::unique_ptr`/`std::shared_ptr` 而非裸指针
- [ ] 析构函数是否 `virtual`（如果类可能被继承）
- [ ] 拷贝构造函数/赋值运算符是否符合"三五法则"（Rule of Three/Five）
- [ ] 移动语义（`&&`、`std::move`）是否正确使用
- [ ] 模板参数是否有必要的 `const`/`noexcept` 约束

### 5.2 C 风格问题（在 C++ 中避免）
- [ ] 是否使用 `malloc`/`free`（应改用 `new`/`delete` 或智能指针）
- [ ] 是否使用 `printf`/`scanf`（应优先用 `iostream`）
- [ ] 是否使用 C 风格类型转换（应改用 `static_cast`、`reinterpret_cast` 等）

### 5.3 现代 C++ 特性
- [ ] 是否使用 `auto` 推导复杂类型
- [ ] 范围 `for` 循环是否可用（避免手写迭代器）
- [ ] `std::optional`/`std::variant` 是否比裸指针/联合体更合适
- [ ] `constexpr` 是否可用于编译期计算
- [ ] `std::array`/`std::vector` 是否比 C 数组更合适

---

## 6. 项目约定检查

### 6.1 文件组织
- [ ] 头文件是否有 `#pragma once` 或传统包含守卫
- [ ] 头文件是否 `#include` 最少依赖
- [ ] 实现文件是否按统一顺序组织（如：`#includes` → `namespace` → 函数定义）

### 6.2 构建与测试
- [ ] 新增代码是否有对应的单元测试
- [ ] 测试覆盖率是否达标
- [ ] 编译警告是否全部解决（无 `unused variable` 等）

---

## 7. 严重级别定义

### 高危
- 运行时崩溃（空指针解引用、数组越界）
- 安全漏洞（缓冲区溢出、注入、信息泄露）
- 逻辑错误（死循环、条件错误）

### 中危
- 内存泄漏/资源泄漏
- 未定义行为（悬垂指针、未初始化使用）
- 边界条件未处理

### 低危
- 代码风格不一致
- 可读性问题（命名、注释）
- 性能优化建议
